# 서비스 자동 시작·종료 관리 개선

> 작성일: 2026-06-15
> 상태: 완료

## 배경

Windows 시작 시 LLM Manager 앱은 자동 실행할 수 있었지만, 앱이 시작된 뒤
`wiki-mcp`, `bgem3-embedding` 같은 개별 서비스를 자동으로 기동하는 정책은 없었다.
또한 PID 파일 기반 복원은 상태만 `RUNNING`으로 표시하고 PID·업타임·메모리 추적은
정확히 복원하지 못했다.

## 완료 항목

| 항목 | 내용 | 주요 파일 |
|------|------|-----------|
| 서비스별 자동 시작 옵션 | `ServiceDefinition.autoStart` 추가. `services.json`에 서비스별로 저장 | `ServiceDefinition.java` |
| 자동 시작 실행 | 앱 초기화 후 `autoStart=true` 서비스만 자동 기동 | `AppContext.java` |
| UI 옵션 | 서비스 설정 탭과 서비스 추가/수정 팝업에 "LLM Manager 시작 시 자동 실행" 체크박스 추가 | `main.fxml`, `add-service.fxml`, `MainController.java`, `AddServiceController.java` |
| 설정 보존 | builtin 서비스 설정 팝업과 상세 수정 흐름에서도 `autoStart` 유실 방지 | `BuiltinServiceSetupController.java`, `ServiceDetailDialog.java` |
| 고아 프로세스 복원 | PID 파일의 프로세스가 살아 있으면 새 프로세스를 중복 기동하지 않고 기존 PID에 연결 | `ProcessManager.java`, `ServiceInstance.java`, `HealthMonitor.java` |
| Windows 종료 강화 | `taskkill /F /T /PID`로 자식 프로세스까지 종료 | `ProcessManager.java` |
| API 노출 | 내장 API 서비스 응답과 OpenAPI 스키마에 `autoStart` 추가 | `EmbeddedApiServer.java` |

## 동작 흐름

### 앱 시작

```
AppContext.init()
  ├─ ServiceRegistry.load()
  ├─ ProcessManager / InstallationService 초기화
  ├─ HealthMonitor.start()
  └─ startAutoStartServices()
       ├─ services.json에서 autoStart=true 서비스 조회
       ├─ 미설치 서비스는 건너뜀
       └─ ProcessManager.start()
            ├─ PID 파일의 기존 프로세스가 살아 있으면 연결
            └─ 없으면 새 프로세스 시작
```

### 앱 종료

```
shutdown hook 또는 AppContext.shutdown()
  └─ ProcessManager.stopAllSync()
       ├─ Process 객체가 있으면 해당 PID 종료
       ├─ Process 객체가 없으면 PID 파일 기준 종료
       ├─ Windows: taskkill /F /T /PID <pid>
       └─ PID 파일 삭제
```

## 대안 검토

### 시작프로그램 등록 명령 갱신

현재 Windows 시작프로그램 등록은 앱 자체 자동 실행만 담당한다.
등록 명령 stale 문제에 대해서는 다음 대안을 검토했다.

| 대안 | 장점 | 단점 | 판단 |
|------|------|------|------|
| 저장 시 항상 레지스트리 값을 현재 명령으로 덮어쓰기 | 설치 경로 변경·업데이트 후 stale command 방지 | 사용자가 수동 편집한 레지스트리 값을 덮을 수 있음 | 권장 |
| 기존 등록 명령과 UI 명령이 다를 때만 갱신 | 불필요한 write 감소 | 기존 명령 조회 파싱 필요 | 가능 |
| "시작프로그램 재등록" 버튼 분리 | 사용자 제어 명확 | 문제를 사용자가 알아야 함 | 보조 기능으로 적합 |

이번 작업에서는 서비스별 자동 시작을 우선 구현했고, 시작프로그램 명령 갱신은 후속 개선으로 남긴다.

### PID 파일 기반 복원

| 대안 | 장점 | 단점 | 판단 |
|------|------|------|------|
| 기존 프로세스에 연결 | 중복 실행 방지, 비정상 종료 복원 가능 | Process 객체가 없어 stdout 재연결은 불가 | 채택 |
| 기존 고아 프로세스를 종료 후 새로 시작 | 로그 연결·상태 일관성 좋음 | 실행 중 작업 강제 중단 위험 | 서비스별 옵션 필요 |
| PID 파일만 정리하고 수동 시작 | 단순 | 자동 시작 경험이 약함 | 비권장 |

채택안은 기존 프로세스를 `ServiceInstance.attachExternalPid(pid)`로 연결하고,
중지는 PID 파일 기반으로 처리한다. stdout/stderr 스트림은 이미 놓쳤기 때문에
재연결하지 않는다.

## 주의 사항

- `autoStart`는 LLM Manager 앱이 시작된 뒤 서비스를 시작하는 옵션이다.
  Windows 로그인 시 앱 자체를 시작하려면 환경설정의 "시스템 시작 시 자동 실행"도 켜야 한다.
- 앱이 정상 종료되면 실행 중 서비스는 종료되고 PID 파일도 삭제된다.
  다음 앱 시작 때 `autoStart=true` 서비스만 다시 시작한다.
- 앱이 비정상 종료되어 PID 파일과 프로세스가 남아 있으면, 다음 실행 때 중복 기동하지 않고
  기존 PID를 관리 대상으로 연결한다.
- `wiki-mcp` 같은 Python 단일 프로세스뿐 아니라 자식 프로세스를 만드는 서비스도
  Windows 종료 시 `/T` 옵션으로 함께 종료한다.

## 검증

- `.\gradlew.bat build` 성공
- FXML 핸들러 연결 확인: `main.fxml`, `add-service.fxml`
- 기존 `services.json`은 `autoStart` 필드가 없어도 Jackson 기본값 `false`로 로드된다.

# TODO — 플러그인 시스템

## 배경 / 핵심 관점

`llm_manager`는 LLM 서비스 정의, 스킬/룰 배포, 프로젝트별 보조 도구가 계속 늘어날 가능성이 높다.
이 기능들을 모두 본체에 직접 넣으면 배포와 설정, 실험 기능 관리가 무거워진다.

초기 플러그인 시스템은 범용 런타임보다 **기능 선언 + 리소스 묶음 + 앱 내부 실행 핸들러 연결**에 초점을 둔다.
외부 코드를 바로 실행하지 않으면 보안, 디버깅, 패키징 난이도를 낮추면서도 플러그인 구조를 먼저 검증할 수 있다.

---

## 목표 구조

```
plugins/
  mes-screen-builder/
    plugin.json
    README.md
    prompts/
    tools/
    assets/
```

`plugin.json`은 플러그인의 식별자, 권한 요청, 기여 지점을 선언한다.

```json
{
  "id": "mes-screen-builder",
  "name": "MES Screen Builder",
  "version": "0.1.0",
  "type": "local",
  "permissions": [
    "filesystem.read",
    "filesystem.write"
  ],
  "contributes": {
    "commands": [
      {
        "id": "mes.createScreen",
        "title": "Create MES Screen"
      }
    ],
    "tools": [
      {
        "id": "generateClxScreen",
        "description": "Generate eXbuilder6 CLX and JS screen files"
      }
    ],
    "settingsTabs": [
      {
        "id": "cursor.settings",
        "title": "Cursor",
        "order": 50,
        "type": "schema",
        "sections": [
          {
            "title": "Environment",
            "fields": [
              {
                "key": "CURSOR_API_KEY",
                "label": "API Key",
                "kind": "env",
                "required": true,
                "secret": true
              }
            ]
          }
        ]
      }
    ]
  }
}
```

## 초기 MVP

### 목표

플러그인을 외부 실행 코드가 아니라 **앱이 읽을 수 있는 선언형 패키지**로 먼저 구현한다.

- [x] 플러그인 ID 중복 검사
- [ ] 플러그인 활성/비활성 상태 저장
- [x] 환경 설정 화면에 `settingsTabs` contribution을 추가 탭으로 표시
- [x] 플러그인 관리 화면에 플러그인 목록 표시
- [ ] `commands`, `tools`, `settingsTabs` contribution을 내부 registry에 등록
  - [x] `settingsTabs` registry
  - [x] `commands` registry
  - [ ] `tools` registry
- [ ] 실행은 앱 내부의 신뢰된 handler에 연결
- [ ] 알 수 없는 command/tool은 실행하지 않고 사용자에게 명확한 오류 표시
- [x] `plugins/` 디렉토리를 배포 패키지에 포함

### 권한 모델

MVP에서는 권한을 실제 샌드박스로 강제하기보다 UI와 실행 전 검증에 사용한다.

| 권한 | 의미 |
|------|------|
| `filesystem.read` | 프로젝트 또는 지정 디렉토리 읽기 요청 |
| `filesystem.write` | 파일 생성/수정 요청 |
| `network` | 외부 API 호출 요청 |
| `process.exec` | 외부 프로세스 실행 요청. MVP에서는 기본 비허용 |

### 완료 기준

- `plugins/` 하위에 새 플러그인 폴더를 추가하면 앱에서 목록으로 확인할 수 있다.
- 잘못된 manifest는 앱 전체를 죽이지 않고 비활성 플러그인으로 표시된다.
- 활성화된 플러그인의 command/tool/settings contribution이 registry에 등록된다.
- 실제 실행은 앱 내부 handler만 수행하며 임의 외부 코드는 실행하지 않는다.

## 1단계 구현 계획: 설정 탭 contribution

### 목표

현재 환경 설정 다이얼로그가 탭 기반이므로, 플러그인이 `settingsTabs` 스키마를 제공하면 메인 앱이 추가 탭을 렌더링한다.
플러그인은 UI 코드를 직접 실행하지 않고 필드 정의만 제공한다.

| `select` | manifest의 `options` 기반 선택 |
| `directory` | 폴더 선택 버튼이 붙는 경로 입력 |
| `password` | 2차 이후 secure storage와 함께 검토 |

### 의존성 설치 정책

- Cursor 플러그인은 `install.scope=global`로 선언하고 `npm install -g @cursor/sdk@latest`로 설치한다.
- 일반 플러그인은 플러그인 디렉토리에 `package.json`이 있으면 로컬 `npm install`을 사용할 수 있다.
- 설치 로그는 플러그인 관리 화면 하단 로그에 스트리밍한다.
- 네트워크 설치 실패는 플러그인 전체 오류가 아니라 의존성 설치 실패로만 표시한다.

### 저장 정책

- `env` 필드는 OS 환경변수 존재 여부만 확인한다.
- secret 값은 `settings.json`에 저장하지 않는다.
- 일반 플러그인 설정은 `AppSettings.pluginSettings[pluginId][key]`에 저장한다.
- 실행 시점에는 플러그인별 allowlist env만 sidecar/프로세스에 주입한다.

## 설계 메모

- MVP는 `lib/def`와 `lib/skills`처럼 파일시스템 기반 루트를 쓰는 방향이 자연스럽다.
- 플러그인 실행을 너무 빨리 열면 보안과 장애 원인 분석이 어려워진다.
- 먼저 manifest와 registry를 안정화하고, 실제 실행 확장은 외부 프로세스형 또는 MCP 서버형으로 분리한다.
